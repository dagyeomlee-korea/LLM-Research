# LLM 서빙 엔진의 무상태성과 문맥 관리 전략

## 1. LLM 서빙 엔진의 무상태성 (Stateless Architecture)

### 개요

vLLM을 비롯한 대부분의 대규모 언어 모델(LLM) 서빙 엔진은 본질적으로 이전 대화를 내부적으로 기억하지 않는 무상태(Stateless) 구조를 가집니다.

### 핵심 특징

#### 독립적인 요청 처리

API 서버로 인입되는 각각의 HTTP/gRPC 요청은 완전히 독립적인 작업으로 간주됩니다.

동일한 사용자가 연속해서 질문을 던져도, 엔진 입장에서는 매번 새롭게 들어온 독립적인 텍스트, 즉 프롬프트일 뿐입니다.

#### 철저한 메모리 격리 (PagedAttention)

vLLM은 다수의 동시 요청을 처리하기 위해 PagedAttention 기술을 사용하여 GPU 메모리(KV Cache)를 효율적으로 할당합니다.

각 요청은 고유한 Request ID를 부여받아 메모리 블록에 매핑되므로, 물리적인 메모리 상에서는 쪼개져 있더라도 논리적으로 완벽하게 격리됩니다.

따라서 다른 사용자의 세션이나 질문 내용이 섞이는 일은 아키텍처상 발생하지 않습니다.

### 요약

LLM 서빙 엔진 자체에는 "기억력"이 존재하지 않습니다. 주어진 입력값(Prompt)에 대해 연산을 수행하고 즉각적으로 결과(Token)를 반환하는 순수한 추론 엔진의 역할만 수행합니다.

## 2. 애플리케이션 계층의 문맥 유지 방식 (Stateful Context Management)

### 개요

사용자가 체감하기에 AI가 이전 대화를 기억하는 것처럼 보이는 이유는, 모델 자체가 아닌 앞단의 웹 서버나 애플리케이션(UI, 미들웨어)이 대화 기록을 능동적으로 관리하기 때문입니다.

### 동작 원리 (Context Injection)

대화의 문맥을 이어가기 위해 애플리케이션은 다음과 같은 파이프라인을 거칩니다.

#### 1. 대화 기록 저장 (History Tracking)

사용자가 채팅을 입력하고 AI가 답변할 때마다, 서버는 해당 세션의 전체 텍스트를 데이터베이스나 인메모리 저장소(Redis 등)에 차곡차곡 저장합니다.

#### 2. 프롬프트 재구성 (Prompt Reconstruction)

사용자가 새로운 질문을 입력하면, 서버는 저장해둔 과거의 대화 기록 전체를 불러옵니다.

그리고 다음과 같은 형태로 하나의 거대한 프롬프트 덩어리를 만듭니다.

```text
[과거 질문 1]
+ [과거 답변 1]
+ [과거 질문 2]
+ [과거 답변 2]
+ [새로운 질문]
```

#### 3. 엔진으로 전체 페이로드 전송

이렇게 구성된 "전체 대화 대본"을 매 API 호출 시마다 vLLM 등의 서빙 엔진에 통째로 전송합니다.

### 요약

AI 서비스의 기억력은 모델의 능력이 아니라 클라이언트/서버의 데이터 주입(Injection) 로직에 의해 구현됩니다. 대화가 길어질수록 모델이 한 번에 읽고 처리해야 하는 컨텍스트 윈도우(Context Window)의 크기도 함께 증가하는 구조입니다.

## 3. 길어지는 LLM 문맥(Context)을 효과적으로 처리하는 4가지 방법

대화형 AI 서비스에서 세션이 길어지면 모델에 전달해야 하는 누적 프롬프트의 크기가 커져 토큰 초과, 응답 지연, VRAM 고갈 문제가 발생합니다.

이를 해결하기 위해 애플리케이션 계층과 서빙 엔진 계층에서 다음 4가지 전략을 조합하여 사용할 수 있습니다.

### 1. 슬라이딩 윈도우 (Sliding Window / FIFO)

가장 직관적이고 구현이 단순한 토큰 관리 기법입니다. 전체 대화 기록 중 가장 최근의 대화(N턴) 또는 특정 토큰 수 이내의 데이터만 프롬프트에 남기고, 오래된 과거 대화는 잘라냅니다(Truncate).

- 구현 방식: Python 리스트 구조에서 `history[-10:]`과 같이 최근 항목만 슬라이싱하여 vLLM API에 전달합니다.
- 장점: 구현이 매우 간단하며, API 호출당 토큰 사용량과 비용의 상한선을 명확하게 통제할 수 있습니다.
- 단점: 지정된 윈도우 범위를 벗어난 과거의 대화 내용은 모델이 완전히 잊어버리게 됩니다.

### 2. 주기적 대화 요약 (Conversation Summarization)

슬라이딩 윈도우의 단점인 문맥 유실을 보완하기 위해, 과거의 대화 내용을 주기적으로 요약하여 프롬프트 상단에 주입하는 방식입니다.

- 구현 방식: 대화 토큰이 일정 임계치를 초과하면 백그라운드에서 요약 프롬프트를 실행하고, 메인 세션 프롬프트를 재구성합니다.
- 장점: 비교적 적은 토큰만 사용하면서도 장기적인 대화의 핵심 흐름과 사용자 의도를 유지할 수 있습니다.
- 단점: 요약 작업을 위한 추가적인 LLM API 호출이 필요하므로 시간 및 비용이 발생합니다.

예시는 다음과 같습니다.

```text
[시스템 프롬프트]
[이전 대화 요약: 사용자는 Proxmox 환경에서 K3s Kubernetes 클러스터를 구축 중이며, 현재 GPU 파드 할당 문제에 대해 논의 중임.]
[최근 대화 3턴]
[새로운 질문]
```

### 3. 의미 기반 검색 (RAG + Vector Database)

전체 대화 기록을 프롬프트에 구겨 넣는 대신, 사용자의 모든 과거 대화를 임베딩(Embedding)하여 벡터 데이터베이스에 저장해 두고 필요할 때만 꺼내 쓰는 진보된 장기 기억(Long-term Memory) 처리 방식입니다.

구현 방식은 다음과 같습니다.

1. 사용자가 새 질문을 입력하면 질문을 임베딩합니다.
2. 벡터 DB에서 현재 질문과 의미적으로 가장 연관성 높은 과거 대화 조각 2~3개를 검색(Retrieve)합니다.
3. 추출된 과거 조각들을 힌트 삼아 현재 질문과 함께 프롬프트에 조합하여 전송합니다.

- 장점: 수개월 전의 대화라도 현재 질문과 관련이 있다면 즉시 참조할 수 있으며, 불필요한 컨텍스트 전송을 없애 토큰을 극도로 절약합니다.
- 단점: 별도의 임베딩 모델 파이프라인과 벡터 DB 인프라 구축이 필요하여 아키텍처가 복잡해집니다.

### 4. 서빙 엔진 최적화 (Serving Engine Optimization)

애플리케이션, 즉 클라이언트 단의 조작 외에도 vLLM과 같은 서빙 인프라 단에서 직접적인 메모리 및 연산 최적화를 수행하여 긴 문맥을 효율적으로 처리할 수 있습니다.

#### Automatic Prefix Caching (APC)

챗봇 환경에서는 "시스템 프롬프트 + 이전 대화 내역(Prefix)"이 매 요청마다 중복 전송됩니다.

vLLM에서 이 기능을 활성화하면, 동일한 Prefix에 대해 이미 계산된 KV Cache를 버리지 않고 GPU 메모리에서 재사용합니다.

이를 통해 연산을 획기적으로 생략할 수 있어 TTFT(Time To First Token)가 대폭 단축되고 전반적인 TPS(Tokens Per Second) 성능이 크게 향상됩니다.

#### KV Cache 양자화 (Quantization)

길어지는 문맥은 GPU VRAM(KV Cache)을 빠르게 점유합니다.

H100, H200과 같은 하이엔드 GPU나 Dell R640 등 로컬 서버의 제한된 메모리 환경에서 FP8 또는 INT8 단위로 KV Cache를 양자화하여 저장하면, VRAM 고갈을 방지하고 동시 처리 가능한 세션 수를 극대화할 수 있습니다.
